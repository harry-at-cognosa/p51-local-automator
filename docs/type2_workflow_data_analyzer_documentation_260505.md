# Type 2 Workflow — Transaction Data Analyzer

**Backend:** `backend/services/workflows/data_analyzer.py`
**Subprocess script:** `scripts/analyze_data.py`
**Date documented:** 2026-05-05

## Pipeline overview

The type 2 workflow has two database steps:

1. **Step 1 — "Analyze data"**: invokes `scripts/analyze_data.py` as a subprocess. The script runs four internal phases and writes up to seven artifact files. The Python orchestrator records each as a `workflow_artifacts` row.
2. **Step 2 — "Analyze findings"**: the orchestrator reads two of the script's markdown outputs, sends them to the LLM, and writes one more artifact (`step5_llm_analysis.json`).

So a maximal run produces 8 artifacts total (7 from the script + 1 from the LLM); a minimal run produces 2 (profile + filtered excel from the script + 0 charts + skipped LLM if no profile/summary).

## Configuration (user_workflows.config)

| Key          | Type        | Notes                                              |
|--------------|-------------|----------------------------------------------------|
| `file_path`  | str         | Required. Absolute path to a CSV or Excel file.   |
| `start_date` | str (Y-M-D) | Optional. Start of date filter.                   |
| `end_date`   | str (Y-M-D) | Optional. End of date filter.                     |
| `days`       | int         | Optional. Used with `start_date` instead of `end_date`. |
| `key_fields` | list[str]   | Reserved; not currently consumed by the script.   |

If both `start_date` and `end_date` (or `start_date` + `days`) are supplied, a **prior period** of equal length is computed automatically and used for comparisons.

## Detection rules (run unconditionally during step 1)

The script auto-classifies columns from the input file. These rules drive what gets analyzed and what charts are produced.

### Date column
A single column is selected. First match wins:
- Pandas dtype is `datetime64[ns]`, OR
- Column name (case-insensitive) contains any of: `date`, `time`, `day`, `period`, `wkend`, AND more than 80% of values parse as a date.

### Amount/value columns (zero or more)
Each column qualifies if:
- Column name (case-insensitive) contains any of: `amount`, `price`, `total`, `revenue`, `cost`, `spend`, `charge`, `discount`, `subtot`, `net`, `gross`, OR
- Dtype is `float64`/`int64` AND median absolute value is between 0.5 and 1,000,000 AND name is not in: `seqn`, `id`, `quantity`, `impressions`.

The first amount column in the resulting list becomes the **primary amount** used by all chart and summary logic.

### Category columns (zero or more)
Each column qualifies if:
- Column name (case-insensitive, underscores treated as spaces) contains any of: `category`, `type`, `channel`, `status`, `account`, `group`, `class`, `department`, `condition`, AND
- 2 ≤ unique values ≤ 500.

The first category column becomes the **primary category** used by chart 1 and chart 3.

### Low-value column drops
A column is dropped from the filtered dataset if:
- It has ≤ 1 unique value, OR
- More than 80% of values are null, OR
- Its name contains: `address`, `bill_addr`, `ship_addr`, OR
- It's an object dtype with mean string length > 80 (likely descriptions or addresses).

## The seven potential script artifacts

Generated by `scripts/analyze_data.py` and recorded as `workflow_artifacts` rows by step 1. Each artifact is conditional on inputs; missing artifacts indicate the relevant condition was not met.

### 1. `step1_data_profile.md` — always produced

A markdown profile of the loaded dataset. Contains:
- Row count, column count.
- Detected date / amount / category columns.
- Date range (if a date column was detected).
- Per-column summary table: column name, dtype, unique count, null count, three sample values.
- Per-amount-column statistics: count, sum, mean, median, min, max, standard deviation.

### 2. `step2_filtered_data.xlsx` — always produced

Excel workbook with the filtered, low-value-dropped data:
- Date-range filter applied if `start_date`/`end_date` were configured AND a date column was detected.
- Low-value columns removed per the drop rules above.
- Header row formatted: blue fill (#4472C4), bold white font, centered.
- Float values formatted as `#,##0.00`.
- Approximate auto-width per column (capped at 40).
- Frozen header row.
- Autofilter enabled.

### 3. `step3_summary_report.md` — always produced

Markdown analysis summary. For each detected amount column, outputs a metrics table (Total, Mean, Median, Min, Max, Count). When a prior period was computed, the table gains two extra columns: prior-period value and percent change.

For each detected category column, outputs a top-15 distribution table (count and percent of period).

### 4. `step3_chart_by_category.png` — conditional

Horizontal bar chart of total primary-amount by primary category. Top 12 categories shown if there are more. Color-graded along a red-yellow-green gradient.

**Generated only if** at least one amount column AND at least one category column were detected.

### 5. `step3_chart_trend.png` — conditional

Trend chart: bars for total primary amount, secondary y-axis line for transaction count, by period.

**Period granularity** is auto-chosen from the data's date span:
- > 365 days: monthly bins
- 61–365 days: weekly bins
- ≤ 60 days: daily bins

**Generated only if** at least one amount column AND a detected date column.

### 6. `step3_chart_comparison.png` — conditional

Side-by-side bars for top 10 categories: prior period (gray) vs current period (blue), using primary amount totals.

**Generated only if** all three: (a) a prior period was computed (i.e., the user supplied `start_date`+`end_date` or `start_date`+`days`), (b) at least one amount column, (c) at least one category column.

### 7. `step4_quality_report.md` — always produced

Markdown data-quality report:
- Missing-values table (per column, count and percent).
- Duplicate-row count (whole-row exact duplicates).
- Per-amount-column outlier section, using a 3-sigma threshold (mean ± 3 × stddev). Up to 20 outlier rows shown for context, plus a "and N more" footer if more exist.
- Date-quality count of unparseable values (if a date column was detected).
- Footer with total issues flagged across all sections.

## The eighth artifact (workflow step 2)

### `step5_llm_analysis.json` — conditional

Produced after step 1 succeeds. The orchestrator reads `step1_data_profile.md` and `step3_summary_report.md`, sends them to Claude with a fixed system prompt, and writes the JSON response. Schema:

```json
{
  "summary": "1-2 sentence overview",
  "findings": ["..."],
  "anomalies": ["..."],
  "suggested_charts": ["chart type: brief description"]
}
```

**Skipped** (with a "Skipped: no profile or summary report found" step note) if neither markdown was produced — currently impossible since both step 1 markdowns are unconditional, but the safeguard exists.

## Failure modes

- `file_path` missing or empty → run fails immediately, 1-step run.
- `file_path` not found on the backend host → run fails immediately.
- Subprocess returncode ≠ 0 → step 1 fails with the first 500 chars of stderr; run fails.
- Subprocess exceeds 120s timeout → exception; run fails.
- LLM call raises → step 2 fails; step 1 artifacts are still recorded.

## What this workflow does NOT do

- Choose or override column detection (no way to force "treat column X as the date").
- Generate charts beyond the three rule-driven ones.
- Persist any summary back to the DB (only files on disk + step output_summary text).
- Process more than one input file per run. (See backlog: "Folder-batch analyzer" workflow type.)
